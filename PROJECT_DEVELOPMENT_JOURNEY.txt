# Homestead Project: A Development Journey

This document outlines the complete development lifecycle of the Homestead project, a full-stack web application for property rental management. It is intended to serve as a guide for understanding the project's architecture, technology choices, and the iterative process of its creation, making it suitable for a final semester presentation.

---

## 1. Project Overview & Vision

**What is Homestead?**

Homestead is a proof-of-concept property management platform, akin to a simplified Airbnb. It empowers users to:
- Register and log in securely.
- Browse and search for property listings.
- View detailed information and photos for each property.
- Book properties for specific dates.
- As a "host," list and manage their own properties, including uploading photos.

The vision was to create a modern, responsive, and intuitive user experience, built on a robust and scalable technology stack.

---

## 2. The Architectural Blueprint: Technology Stack Rationale

Choosing the right tools is the foundation of any successful software project. For Homestead, we selected a popular, JavaScript-based stack, often referred to as a "MERN" stack variant. This choice was deliberate, prioritizing development speed, a unified language across the stack, and a vast ecosystem of community support.

- **Backend: Node.js & Express.js**
  - **Why?** Node.js allows us to use JavaScript on the server, creating a seamless development experience with the frontend. Its non-blocking, event-driven architecture is highly efficient for I/O-heavy applications like a web server that constantly handles requests to a database. Express.js is a minimalist and flexible framework for Node.js that provides a robust set of features for building web and mobile applications, simplifying the process of creating a RESTful API.

- **Frontend: React (with Vite)**
  - **Why?** React is the industry standard for building dynamic, component-based user interfaces. Its "write once, reuse anywhere" component model accelerates development and simplifies state management. We chose **Vite** as our build tool over the older Create React App because of its near-instantaneous Hot Module Replacement (HMR) and faster build times, leading to a significantly improved developer experience.

- **Database: MongoDB (with Mongoose)**
  - **Why?** As a NoSQL database, MongoDB's flexible, document-based data model (storing data in JSON-like objects) is a natural fit for JavaScript. It allows for an evolving schema, which is ideal during development. We used **Mongoose** as an Object Data Modeling (ODM) library; it provides a straightforward, schema-based solution to model application data, including type casting, validation, and business logic hooks.

- **Styling: Tailwind CSS**
  - **Why?** Instead of writing custom CSS files, we chose Tailwind, a utility-first CSS framework. This allowed us to build custom designs directly in our markup by composing utility classes (e.g., `flex`, `pt-4`, `text-center`). This approach drastically speeds up UI development, enforces design consistency, and keeps the project's CSS footprint minimal.

- **Authentication: JSON Web Tokens (JWT)**
  - **Why?** For user authentication, we implemented a token-based system using JWT. It is a stateless, secure, and widely adopted standard. When a user logs in, the server creates a signed token containing user information and sends it to the client. The client then includes this token in the header of subsequent requests to access protected routes.

## 3. The Development Journey: An Iterative Process

The development of Homestead was not a linear path but an iterative journey of building, testing, and refining. The `project_summary.txt` log was instrumental in documenting this process, which can be broken down into three key phases.

### Phase 1: Backend Foundation
The first step was to build the "engine" of our application.

1.  **Server & API Scaffolding:** An Express server was initialized, and a clear, modular folder structure was created (`routes`, `controllers`, `models`).
2.  **Data Modeling:** Using Mongoose, we defined the core schemas for our application: `User`, `Property`, `Booking`, and `Review`. These schemas act as the blueprint for our data.
3.  **API Endpoint Implementation:** We built out the RESTful API endpoints for all CRUD (Create, Read, Update, Delete) operations. This included routes like `POST /api/auth/register`, `GET /api/properties`, and `POST /api/bookings`.
4.  **Security Implementation:** From the start, security was a priority. We integrated `bcryptjs` to hash user passwords before saving them to the database and implemented JWT for session management. Middleware like `cors`, `helmet`, and `xss-clean` were added for further protection.

### Phase 2: Frontend Construction
With the backend API taking shape, we began building the user interface.

1.  **Project Initialization:** A new React project was scaffolded using Vite.
2.  **Routing:** Using `react-router-dom`, we defined the application's URL structure, creating routes for public pages (Home, Login, Property Details) and protected pages (Dashboard, My Bookings).
3.  **Component-Based UI:** We broke down the UI into a hierarchy of reusable components (`Navbar`, `PropertyCard`, `BookingForm`, etc.), which is a core tenet of React development.
4.  **State Management:** React's Context API was used to manage global state, such as the currently logged-in user (`AuthContext`), making user data accessible throughout the application without "prop drilling."
5.  **API Integration:** We created a dedicated `services` layer using the `axios` library to handle all communication with our backend API. This keeps our component logic clean and centralizes API calls.

### Phase 3: Integration, Debugging, and Deployment
This phase was focused on making the frontend and backend work together seamlessly and getting the project live. It involved overcoming several critical challenges.

1.  **Challenge: Deploying a Monorepo to Vercel.**
    - **Problem:** Our initial Vercel deployment failed with 404 errors on all pages except the homepage.
    - **Solution:** This is a classic Single-Page Application (SPA) issue. We configured a `rewrites` rule in `vercel.json` to redirect all non-API requests to `index.html`. This allows React Router to take control of the navigation on the client-side. We also learned to properly set the "Root Directory" to `frontend` in the Vercel UI, leveraging its framework-aware build system instead of over-complicating the configuration file.

2.  **Challenge: The Local File Upload Fiasco.**
    - **Problem:** The backend server was crashing due to a Git merge conflict that left invalid text (`<<<<<<< HEAD`) in the code. After fixing this, image uploads were still not working correctly.
    - **Solution:** The conflict was between cloud (AWS S3) and local file upload logic. We standardized on the local logic for development. The core issue was a mismatch in `multer`'s storage strategy: it was holding files in memory (`memoryStorage`) while the controller expected a file on disk (`diskStorage`). We corrected the middleware to use `diskStorage`, saving files to the `backend/public/uploads` directory.

3.  **Challenge: Broken Images in the Browser.**
    - **Problem:** Even with uploads working, images were broken on the site.
    - **Solution:** The frontend (e.g., `localhost:5173`) was requesting images from a relative path, but the files were being served by the backend (`localhost:5000`). We fixed this by programmatically prepending the backend's base URL to the image paths in our React components, ensuring the browser fetched them from the correct source.

## 4. How to Run This Project Locally

To run Homestead on your local machine, follow these steps:

1.  **Clone the Repository:**
    ```bash
    git clone <repository-url>
    cd Homestead
    ```

2.  **Setup the Backend:**
    ```bash
    cd backend
    npm install
    # Create a .env file based on .env.example and fill in your database connection string, etc.
    npm run dev 
    # The backend server will start, typically on http://localhost:5000
    ```

3.  **Setup the Frontend (in a new terminal):**
    ```bash
    cd frontend
    npm install
    npm run dev
    # The frontend development server will start, typically on http://localhost:5173
    ```

4.  **Access the Application:**
    - Open your web browser and navigate to `http://localhost:5173`.

## 5. Conclusion

The Homestead project successfully meets its goal of being a functional prototype for a property rental platform. The journey from initial setup to a stable, deployable application was filled with valuable learning experiences, particularly in debugging the intricate interactions between a separate frontend and backend, and in navigating the specific requirements of both local and cloud environments. The chosen tech stack proved to be robust, efficient, and a pleasure to work with, validating our architectural decisions.
